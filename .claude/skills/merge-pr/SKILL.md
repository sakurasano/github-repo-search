# PRをマージして後始末する

作業ブランチのPRをマージするときの一連の手順。目的は、CIが通っていない変更を誤って取り込まないこと、
そしてマージ後に散らかる不要ブランチとローカルmasterの遅れを毎回もれなく片付けること。

masterはbranch protectionで保護され、CI(`Test & Lint`)の通過とmaster最新への追従が必須条件になっている（管理者も対象）。
条件を満たさないマージはGitHubが拒否するので、下記の確認は拒否される前に気づいて分かりやすく伝えるためのもの。

このリポジトリはsquashマージのみ許可されている。マージ方式は`--squash`で固定する。

## 手順

### 1. 前提を確認する

- 今いるブランチを確認する（`git branch --show-current`）。masterなど既定ブランチ上ならマージ対象がないので、その旨伝えて中断する。
- 未コミットの変更があれば（`git status --porcelain`）、意図しない取りこぼしを防ぐため先に伝えて確認する。
- ブランチに対応するPRを取得する。開いているPRが無ければマージできないので中断して伝える。`mergeStateStatus`が`BEHIND`ならmasterに追従していないので、`gh pr update-branch`で更新してから進む（更新でCIが再実行される）。

```bash
gh pr view --json number,title,state,mergeStateStatus
```

### 2. CIを確認する（マージ前の必須ゲート）

ここが最重要。CIが緑でなければ絶対にマージしない。`gh pr checks`の終了コードで判断する。

```bash
gh pr checks
```

- 終了コード0（すべて通過）→ 次へ進む。
- 終了コード8（保留中＝まだ実行中）→ 勝手にマージせず、ユーザーに「まだCI実行中。完了を待つか？」と確認する。待つ場合は`gh pr checks --watch`で完了まで見届け、結果を再判定する。
- 終了コード1（失敗あり）→ マージしない。どのチェックが落ちたかを示して中断する。

失敗・保留中のときにゲートを黙って通すのは、このスキルの目的そのものに反する。判断を勝手にせず必ずユーザーに返す。`--admin`でゲートを回避してはいけない。

### 3. マージ内容を提示して確認を取る

マージとブランチ削除は取り消しにくい操作なので、実行前に何をするかを一度提示して了承を取る。提示するのはPR番号・タイトル・マージ先・「squashマージ＋ローカル/リモートのブランチ削除＋master更新を行う」という要約。

### 4. squashでマージし、ブランチを削除する

```bash
gh pr merge --squash --delete-branch
```

`--delete-branch`がローカルとリモート両方のブランチを削除し、既定ブランチ（master）に自動で切り替える。

### 5. ローカルのmasterを最新化する

squash後のコミットをローカルmasterに取り込む。

```bash
git checkout master   # 既に切り替わっていれば不要
git pull
```

### 6. 結果を報告する

マージしたPR、削除したブランチ（ローカル/リモート）、更新後のmasterの位置（`git log --oneline -1`）を簡潔に伝える。

## 中断する場合の原則

CI未通過・PR不在・master上・未コミット変更などで中断したときは、何が起きていて次に何をすればよいかを伝える。
部分的に進めて中途半端な状態を残さない（例：CIが落ちているのにマージだけ進める、は不可）。
